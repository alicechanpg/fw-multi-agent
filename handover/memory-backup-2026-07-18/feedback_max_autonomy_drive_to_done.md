用戶多次強調（2026-07-01）：要我**想清楚自己能做到什麼、能推進到多遠，然後就做到那個程度**，不要反覆停下來問或丟選項回去。

**Why:** 這個 session 我一直 defer（「要 A 還 B」「你覺得呢」「要我盯嗎」），用戶連續回「不要再問我」「甚麼 A 還 B」「你要想你可以做的事情 做多到甚麼程度 記住」。他要的是能自己判斷、自己執行到底的協作者。

**How to apply:**
- 需求有歧義但能從證據/慣例推定時 → 自己決定、講一句假設、往下做，不要開選項問。
- 遇到「硬限制」時，先想「我還能做什麼繞過」（例：Jenkins BRANCH 是 Choice → 直接改 config.xml 加 branch 觸發完再還原/整理），而不是把限制當終點丟回給用戶。改共用設定這種有 blast radius 的事，做之前用證據確認安全（backup、驗 XML、確認別人沒在用），做完驗證，而不是不敢做。
- 「做到完」= 推到**已驗證**（build 綠、測試過），不是「觸發了/送出了」就回報。失敗要查 root cause 再報，不是只說「失敗了」。
- 完成 milestone 後主動接續下一步（回報到 PR、handoff、update 狀態），參考 [[feedback_dont_stop_to_ask]]。
- 內部 subagent 討論（L1 鐵律）≠ 問用戶；該做的內部審核照做，但不要拿決定去煩用戶。

**但要平衡（2026-07-01 用戶補充）：不是悶頭全做，而是——**
- 非 trivial 任務**先講規劃再動手**：一句話講清楚要做什麼、順序、預期產出。
- 附一份**分工**：明確分「我自己做（做完回報）」vs「要你 approve 才做」兩桶。把 blast radius 大的、對外的、不可逆的放進 approve 桶。
- 自主桶的事直接做到已驗證完成；approve 桶的事先備好（證據、確切 end-state），只等你點頭。

## 2026-07-17 加強：**技術決定一律歸自主桶**

用戶原話：「**我看不懂 code 但是我很知道我的需求是甚麼**」「**如果技術你幫我決定就好了**」。

**Why:** 那個 session 我把 code-level 的取捨開成選項丟回去問了六次（要不要修 `_is_date`、unused param 留不留、substring 比對要不要硬化、`lookup()` 要不要加參數…）。用戶每次都回「建議」——等於只是幫我按同意鍵。他不讀 code，這些問題對他沒有資訊價值，只是把我的工作丟回給他。

**How to apply:**
- **實作層的取捨永遠不要問**：函式簽章、參數、演算法、要不要修某個缺陷、測試怎麼寫、偏不偏離計畫的參考程式碼——自己判斷、自己修、做完講結果。
- **approve 桶只留「用真實世界後果」能講清楚的事**：force-push、刪不可回復的檔案、對外送出的訊息、要花錢的、動到別人在用的東西。判準是「用戶不看 code 也能判斷這題」——若不能，就是我的題。
- **回報用需求語言，不是 code 語言**：講「這會讓一條早該過期的事實永遠被當成真的」，不是講「`fromisoformat` 接受 ISO basic format」。細節收進 details 區塊給想看的人。
- 這條不減弱 [[feedback_evidence_based]]：技術決定我自己做，但**每個結論仍必須用工具驗證**，不能因為「反正他不看」就鬆手。相反地，因為他不看，我是唯一的把關者。
