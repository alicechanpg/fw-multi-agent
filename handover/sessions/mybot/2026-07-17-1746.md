## ⭐ 已實測結論（上機量到的，非推測）

### 1. Audio frame 預算本來就很緊 — 閒置就吃掉 66%
```
[GP]  audio_frame  n=1391  avg=479us  max=540us  over_budget=0
```
預算 **725µs**（32 frames @ 44.1kHz）。`n=1391/s` ≈ 理論 1379 → probe 校準正確。**閒置 avg 479µs = 66%，尖峰 74%，只剩 ~185µs 餘裕。**

### 2. Knob 路徑完全不影響 audio（對照組）
`app_evt=0`、`audio_overrun=0`、`over_budget=0`、`audio_frame` 與閒置**完全相同**。
且 `vol_step_max` 到 **320/1000**（0.32 的大跳階）仍無 glitch → **證明 VolumeRamp 吃得下大跳階，「跳太大」不是 glitch 成因。**

### 3. ⭐ SPI 指令流量 → audio miss deadline（因果同步，實測）
```
ble_status = 0x1                                    ← BLE 連上
[GP] audio_overrun=3   over_budget=1  max=748us     ← 破 725µs
[GP] spi_process_cmd   max=2146us                   ← baseline 才 ~550µs
get_main_fw_version / get_serial_number / get_main_fw_version
[GP] audio_overrun=17  over_budget=9  max=751us     ← 一秒 miss 17 次
[GP] spi_process_cmd   max=3489us                   ← 3.5ms
ble_status = 0x0                                    ← BLE 斷
[GP] audio_overrun=0                                ← 立刻恢復
```
→ **使用者原假設「SPI task 耗時太久」成立；glitch 是 dropout 不是 click。**
→ ⚠️ **但這是 BLE 連線時的查詢 burst，不是 app 調音量本身**（`app_evt` 全程 0，log 中無 `set_inout_volume`）。同一條 SPI 路徑，高度相關但尚未直接量到 volume 指令。

### 4. ⭐⭐⭐ ROOT CAUSE：SPI6 中斷模式 → ISR 爆發 → 搶佔 Realtime7 audio task

**共同點不是「誰送的」也不是「送什麼」，是「SPI6 上有沒有流量」。**

| 操作 | SPI6 流量 | Glitch | `audio_isr_stolen` |
|---|---|---|---|
| 轉 knob | **零** | ❌ 無 | avg=0 max=0 |
| App 拖音量 | 28 筆/秒 (RX) | ✅ 連續 | avg=13-16µs **max=191µs** |
| **HW 切 preset** | TX passive cmd → **app 回推 ~250 筆參數** | ✅ 破音 | 同上 |

> **HW 切 preset 的真實流程**（使用者確認）：實體切換 → STM32 送 passive cmd 給 app → **app 收到後把整個 preset 的參數用 `set_dsp_param` 一個一個推回來**（實測 9 秒 ~250 筆，`app_evt=22-30/s`）→ 破音發生在這個來回。
> ⚠️ 注意：`switch_preset_callback` **從未被呼叫**（它自己的 DEBUG_OUT 沒出現）→ App 切 preset 不送 `switch_preset` 指令，而是推參數流。

**機制**：`SPI_Command_Master.cpp:120/142/167` → `HAL_SPI_TransmitReceive_IT(&_spi, ..., SPI_MAX_BUFFER_SIZE)`，`SPI_MAX_BUFFER_SIZE=520`（`SPI_Command_Master.h:14`）。全專案**無 `hdma_spi6`** → SPI6 完全沒配 DMA。中斷模式下每個 FIFO 事件一次 ISR，一筆 520B = 幾十上百次 ISR。**ISR 是唯一能搶佔 Realtime7 的東西。**

**餘裕不足是幫兇**：閒置 `audio_frame avg=482µs` / 預算 725µs = **66%**，只剩 243µs。單次 ISR 爆發偷 191µs = 吃掉 **79%** 餘裕。

## 🔧 修法：SPI6 改用 BDMA（**下一步的主線**）

520 bytes 從上百次 ISR 降到 **1 次**。可行性已全部查證：

| 疑慮 | 查證結果 |
|---|---|
| ESP32 端要配合改嗎？ | **不用**。`pg-spark-esp32-wifi-bt/components/spi_command/SPI_Command_Slave.cpp:47` → `spi_slave_initialize(..., SPI_DMA_CH_AUTO)` — **ESP32 早就在用 DMA**。且 master 的搬運方式（polling/IT/DMA）對 slave 不可見，線上只有 clock+data。 |
| 「designer 不知道 ESP32 怎麼做」（原始原因） | **不成立**，見上。 |
| SPI6 在 D3 domain，只能用 BDMA | BDMA 8 個 channel，**只有 0/1 被 SAI4 佔**（`hdma_sai4_a` 等）→ **2-7 全空** |
| BDMA 只能存取 D3 SRAM | Linker **已有** `RAM_D3 : ORIGIN=0x38000000, LENGTH=64K`（`.ld:50`）+ 現成 `.dma_buffer_RAM_D3` section（`.ld:245`） |
| D3 SRAM 夠嗎 | SAI4 buffer 只用 ~1KB/64KB → 520B 綽綽有餘 |

**要改的 4 件事**：
1. `bdma.c`/`spi.c`：加 SPI6 的 BDMA channel（2-7 擇一）+ IRQ handler
2. `SPI_Command_Master`：tx/rx buffer 移入 `.dma_buffer_RAM_D3`，**32-byte 對齊**
3. `HAL_SPI_TransmitReceive_IT` → `_DMA`（3 處：`:120, :142, :167`）
4. **Cache 維護**：TX 前 `SCB_CleanDCache_by_Addr`、RX 後 `SCB_InvalidateDCache_by_Addr`（照抄 SAI4 那條路的作法）

⚠️ **依 repo 規則 `fw-workflow.md`「Modify ISR/DMA → Design Review: Yes」**，且 **FWP-364 剛重寫過這塊 SPI**（`8a61b5c`）→ 建議先在 debug branch 做出數據當 review 材料，再走正式流程。
