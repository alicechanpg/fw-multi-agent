## JIRA comment 草稿 (待發布到 FWP-809)

### Round 3 Stress Test — btController WDT Crash 觸發，調整策略

#### Crash 事件

Round 3 第 4 cycle (前 3 cycle PASS) 出現 ESP32 BT Controller task watchdog timeout。

| 項目 | 數據 |
|------|------|
| 發生位置 | ESP32 OTA 40%→50% 之間 |
| 卡死的 task | btController（Espressif closed-source binary） |
| Backtrace | `btdm_controller_task` → `ble_try_turn_on_pll_track` → `rw_schedule` |
| 持續時間 | 3.5 分鐘未恢復，BLE 斷線、OTA 停止 |

重點：三個函數全在 Espressif 不公開原始碼的 BT controller binary 裡，不是我們的 application code 問題。

#### 為什麼改 latency

ESP-IDF v5.3.2 修了 "BLE channel map update issue when latency is non-zero" 的 bug。我們目前用 v5.0.4（已 EOL），OTA 期間的 conn params 用了 latency=4，剛好命中。

latency（Slave Latency）= BLE 裝置可以連續跳過幾次心跳不回應，用來省電。但 latency>0 會讓 BT controller 的 radio 排程邏輯變複雜。在 WiFi OTA 高流量搶 radio 的壓力下，更容易觸發 controller binary 裡面的排程 bug。

決策：
- latency=4 → latency=0（消除 controller 排程複雜度，避開已知 bug 類別）
- interval 維持 500-1000ms（減少 BLE 空封包頻率，節省 radio time 給 WiFi）
- timeout 維持 20s（超過 OTA progress gap ~11s，不會 supervision timeout）

#### Round 3 修正後的參數

| 參數 | 原始值 | Round 2 (OTA期間) | Round 3 (OTA期間) |
|------|--------|--------------------|--------------------|
| min_int | 0x10 (20ms) | 0x60 (120ms) | 0x190 (500ms) |
| max_int | 0x20 (40ms) | 0xC0 (240ms) | 0x320 (1000ms) |
| latency | 0 | 4 | 0（從 4 改回 0，避開 controller bug） |
| timeout | 400 (4s) | 600 (6s) | 2000 (20s) |

#### 測試計畫

用上述參數跑 50+ cycles stress test，觀察：
1. btController WDT crash 是否消除
2. BLE 斷線率是否進一步降低
3. 500-1000ms interval + latency=0 在 Apple BLE guidelines 下是否穩定
