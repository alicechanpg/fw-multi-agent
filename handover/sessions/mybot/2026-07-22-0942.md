## 本次完成（結論：非 SA-1497 length bug 重現；風險在 MAC「值」的時序）
用 code 驗證整條鏈（file:line 皆已核對）：

1. **Mini2 的 ESP32 firmware = `pg-spark-esp32-wifi-bt`**（commit `cb91660`「advertise as
   Gen2 so App can find MINI2」+ `SPI_Command_Slave.h` 的 `DEV_SPARK_MINI2`）。
2. **SA-1497 length fix 已在**：`components/ble/ble.c:57` array 11 bytes `{29 5d 02 22 FF×6 00}`，
   `ble.c:67` `manufacturer_len = sizeof(manufacturer) - 1` = **10** → 廣播 `29 5d 02 22 <MAC×6>`，
   尾端 `00` 已排除。→ 這**不是** SA-1497 那個 length bug 重現。
3. MAC 來源 = **STM32**（非 ESP32 自身 BT stack）：`app_general_process.cpp:110`
   `send_msg_hwcmd_get_bt_mac_address(&manufacturer[4])`，開機後 `vTaskDelay(2000)`（:108）**只問一次**。
4. ESP32 拿到 → 寫 NVS → `esp_restart` → 下次 `ble_init` 讀 NVS 廣播（`app_general_process.cpp:125/134/137`、`ble.c:1333-1337`）。
5. STM32 回的 MAC = WB2853 audio 晶片的 `_mac`（`BleCmdHandler.cpp:2366` → `WB2853.h:170 get_mac_string`）。
6. `_mac` **只**靠 WB2853 非同步 UART 交握填入：晶片 `BOOT_FINISH`→`get_mac_addr()`（`WB2853.cpp:620`）
   →晶片回 `0xA7`→`_mac[i]=buf[4+i]`（:665-667）；`AudioController.cpp:91` 的主動抓取 **被註解掉**。

## 根因假說（待驗證，非事實）
`_mac` 填入相依 WB2853 開機 + UART 交握；ESP32 在「開機 +2s」只問一次。若那時 WB2853 還沒回 MAC，
ESP32 會把 **0 或舊值** cache 進 NVS 並持續廣播 → iOS App 配不到 audio → 「Connect to Spark Audio」。
病灶類別同 SA-1497（「ESP32 存的 BT MAC 是 \0\0\0\0\0\0」），但這次是**時序**、非 length。
- **驗證方法**：故障機上 (a) BLE sniffer/nRF Connect 讀 `0x5D29` 6 bytes；(b) 對照 STM32 debug UART
  的 `0xA7` MAC reply（`WB2853.cpp:668` 有 DEBUG_OUT）。兩者比對真實 A2DP 位址即可證實/推翻。
- **信心**：待驗證（iOS App 比對邏輯 closed-source，屬行為觀察）。
