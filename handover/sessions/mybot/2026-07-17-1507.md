### 1. app vs knob(hw) 路徑差異（已驗證，附 file:line）

| | Knob (HW) | App |
|---|---|---|
| 入口 | InputService ADC → event flag | ESP32 → SPI6 → SysEx parse |
| Task | `vol_ctrl_task` `AudioController.cpp:265`，**Normal1**，event 驅動 | `spi_command_process_task` `BleCmdHandler.cpp:80`，**AboveNormal**，polling 每 5–10ms |
| 呼叫 | `set_input_volume(ch, v, from_hw=true)` `:296,:304` | `set_inout_volume_callback` `:881` → `set_input_volume(ch, v)`，`from_hw` **預設 false**（`AudioController.h:414`）|
| 回 notify app | 會（mixerview 開時 post `OBJ_OUTPUT_VOLUME`）`:788-798` | 不會 |
| 套 gain | 寫 `input_vol[ch]` → 同一個 `VolumeRamp` | **完全相同** |

**承重結論**：`set_input_volume`（`AudioController.cpp:755-799`）只寫一個 float 陣列，沒碰 I2C/DSP。實際套 gain 在 `audio_process_task` 的 `VolumeRamp`（`AudioService.cpp:83-109`），指數平滑（每 frame 收斂 2%）+ per-sample 內插，**兩條路徑共用**。
→ **glitch 不是音量值/ramp 造成的**，差異只可能在 SPI 那條路徑「順便做的其它事」。preset 也走 SPI，共同點成立。

### 2. 使用者的「SPI task 耗時太久」假設有矛盾點（已驗證）
`saiTask` = **osPriorityRealtime7**（`AudioService.cpp:165`），`spiCmdTask` = **AboveNormal**（`BleCmdHandler.cpp:59`），`configUSE_PREEMPTION=1`（`FreeRTOSConfig.h:62`）。
→ **SPI task 單純耗時搶不走 audio task 的 CPU**，會被直接 preempt。要造成 glitch 必須有別的機制。

### 3. 候選機制（**尚未驗證，不可當結論**）
- **XIP 卡死**（目前最能同時解釋 volume + preset）：`STM32H750IBTX_FLASH.ld:46` `FLASH (rx) : ORIGIN = 0x90000000` = 外部 QSPI NOR；`AudioService.o`／`BleCmdHandler.o` **都不在 ITCM 清單**（只有 25 個 DSP .o 被搬進 ITCM）。任何 QSPI 操作或 I-cache 被洗掉 → audio task 抓不到指令 → 連 Realtime7 都 stall。
  - **缺口**：尚未驗證 app *volume* 路徑到底有沒有碰 QSPI。preset 路徑（`switch_to_preset_slot` / `save_preset` → FatFs）較可能。
- 長 critical section / 關中斷
- SPI DMA 與 SAI DMA 的 bus 競爭

### 4. 兩個已被自己推翻的假設（**別再重查**）
- ❌「app 路徑有 `DEBUG_OUT(%f)`（`BleCmdHandler.cpp:883`）、knob 的 `ACTRL_DBG` 被編掉，所以只有 app 阻塞」→ **錯**：`AudioController.cpp:32` 有 `#define ENABLE_ACTRL_DEBUG`，`ACTRL_DBG` = `DEBUG_OUT`，**兩條都會 log**。
- ❌「`get_input_volume` 的 `index = input_vol/0.00778f` 是 uint8_t 且無 bounds check → 溢位讀爆表」→ **錯**：`VOLUME_TAPER_TOTAL_LEVEL=129`（`AudioController.h:61`），vol=1.0 → index=128 界內；兩條路徑都有 clamp（`BleCmdHandler.cpp:889`）。
  - ⚠️ 但**確實是潛在風險**：`get_input_volume`（`AudioController.cpp:1018`）無 bounds check，而 `test_factory_test.cpp:257` 有。目前靠上游 clamp 保護。未修（使用者規則：不自己修無關的 bug）。
